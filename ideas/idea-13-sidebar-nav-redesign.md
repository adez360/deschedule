>
> 對應元件：`frontend/src/components/shared/app-sidebar.tsx`

## 1. 現況盤點

| 分組 | 項目（順序） | 性質 |
| --- | --- | --- |
| **個人** | 個人資料 → 排班時段 → 門市偏好 → 我的班表 → 薪資報表 | 帳號設定 / 排班輸入 / 排班輸入 / 結果檢視 / 結果檢視 |
| **管理** | 班表管理 → 人員管理 → 門市管理 → 身份組與權限 | 排班作業 / 人員作業 / 組織設定 / 組織設定 |
| **系統設定** | 人力需求 → 截止日設定 | 排班參數設定 / 排班參數設定 |

權限可見性（維持不變）：
- `個人`：所有登入者
- `管理`：`store.schedule.edit` / `org.schedule.arrange` / `org.schedule.view_all`（班表、人員）+ `org.manage` / `system.all`（門市、身份組）
- `系統設定`：`org.manage` / `system.all`

---

## 2. 問題分析

1. **「個人」組順序與使用頻率相反**：員工最常看的是「我的班表」，卻排在第 4；「個人資料」屬於低頻帳號設定，卻排第 1。
2. **輸入與結果交錯**：排班時段、門市偏好（員工*輸入*排班意願）與我的班表、薪資報表（*檢視*結果）混在一起，沒有分層。
3. **`管理` 與 `系統設定` 切分武斷**：兩組都是 `org.manage` 等級的組織設定，但「門市管理」放在 `管理`、性質相同的「人力需求」卻放在 `系統設定`。使用者無法預期某項設定會在哪一組。
4. **排班參數與排班作業被拆開**：人力需求、截止日設定其實是「班表管理」的前置設定，卻被丟到最底下獨立的 `系統設定`，操作時要兩頭跑。
5. **命名不一致**：同一個「班表」概念出現三種詞 — 排班時段 / 班表管理 / 我的班表（排班 vs 班表混用）。

---

## 3. 提案：依「領域 + 使用頻率」重新分組

依「個人自助 → 排班領域 → 組織設定」三層重整，組內依使用頻率（高→低）排序，並把排班參數收攏到排班作業旁邊。

```
個人
  ├─ 我的班表        /schedules      （最高頻，置頂）
  ├─ 排班時段        /availability
  ├─ 門市偏好        /preferences
  ├─ 薪資報表        /payroll
  └─ 個人資料        /profile        （低頻帳號設定，置底）

排班管理            ← 原「管理」的班表 + 原「系統設定」全部
  ├─ 班表管理        /schedules（管理視圖）
  ├─ 人力需求        /settings/demand
  └─ 截止日設定      /settings/deadline

組織設定            ← 原「管理」的人員/門市/身份組
  ├─ 人員管理        /employees
  ├─ 門市管理        /settings/stores
  └─ 身份組與權限    /settings/role-groups
```

**改動重點**

- **個人組重排**：我的班表置頂、個人資料置底，符合查看頻率。
- **消滅「系統設定」這個通用桶**：人力需求 + 截止日設定（排班參數）收進「排班管理」，緊鄰班表管理，設定與套用一氣呵成。
- **「管理」拆成兩組**：排班領域（排班管理）與組織治理（組織設定）分開，分組標準一致 — 一組是「排班怎麼跑」，一組是「組織有誰、有哪些店、誰能做什麼」。
- **權限可見性沿用現有邏輯**：
  - `排班管理`：原 `管理`(班表) ∪ `系統設定` 權限 → 排程相關權限或 `org.manage`/`system.all`
  - `組織設定`：人員 = schedule/manage 權限；門市・身份組 = `org.manage`/`system.all`
  - 需確認：當管理者只有排班權限、無 `org.manage` 時，「組織設定」組只顯示「人員管理」一項（門市/身份組隱藏）。沿用現況的逐項過濾即可。
