# 自動排班邏輯
自動排班會根據可用時段，偏好權重自動安排

## 範例
小明
- 門市A: 0.8
- 門市B: 0.2

小華
- 門市A: 0.3
- 門市B: 0.4
- 門市C: 0.3

1. 假如門市A和門市B在同一時間段都有缺，小明會被排到偏好比重叫高的門市A
2. 假如門市B缺人，小明與小華同時段都可以安排，會優先安排比重比較高的小華

---

## 現況與落差（2026-06-11 分析）

目前排班器（`scheduler.py`）是**逐門市獨立**執行：
- ✅ 規則 2 已成立：同店內候選人依該店偏好權重降冪挑選
- ❌ 規則 1 做不到：單店排班看不到其他門市的需求，無從比較「去 A 還是去 B」
- ❌ **跨店重複指派 bug**：兩家店各自產生班表時，同一員工可能在同一時段被排進兩家店（無跨店互斥）

無論選哪個方案，跨店互斥都必須補上。

## 決策面向

### A. 排班範圍（核心決策）

| 方案 | 說明 | 優點 | 缺點 |
|------|------|------|------|
| A1 全組織聯合排班 | 一次觸發排所有門市；逐時段把「員工×門市」一起比較，員工去自己比重最高的缺人門市 | 完全符合範例語意；天然解決跨店互斥 | 改動較大：觸發入口、API、UI 都從「單店」變「全組織」；單店重排語意要重新定義 |
| A2 維持單店排班 + 全域佔用檢查 | 排某店時，把該員工「他店已排時段」視為不可用 | 改動小；修掉重複指派 | 規則 1 仍不成立——先排的門市搶贏，結果依排班順序而異 |
| A3 單店觸發、幕後聯合 | UI 仍按單店按鈕，但後端重排該組織當週所有 draft 班表 | 介面不變、語意正確 | 按一家店的按鈕卻動到其他店的草稿，違反直覺；已手動微調的他店草稿可能被覆蓋（is_manual 可保護，但自動排部分會變） |

### B. 「同時段多店都缺」的裁決規則（A1/A3 才需要）

- B1 **比重優先**：員工固定去自己比重最高的缺人門市（範例語意）。風險：熱門店吸走所有人，冷門店永遠缺人
- B2 **覆蓋率優先**：以全組織總缺口最小為目標，比重當 tie-breaker。整體供需較好，但個人可能被排到比重低的店（違反範例）
- B3 **混合**：先依比重分配，結束後對仍有缺口的店做第二輪「借調」（只動還有剩餘可用時段的人）

### C. 平手規則（比重相同時）

- C1 缺口大的門市優先拿人
- C2 依門市建立順序／固定順序（可預測、可重現）
- C3 隨機（公平但不可重現）

### D. 既有班表與手動指派的處理

- D1 重排時保留 `is_manual = true`（現行行為），自動指派部分重算——聯合排班時他店的手動指派同樣保留並佔用該員工時段
- D2 published 班表是否參與聯合重排？建議：只重排 draft，published 視為固定佔用

### E. 演算法實作

- E1 greedy 擴展：把現有 day×hour 迴圈外移一層為「組織級」，每時段對所有門市的缺額做配對。改動可控，與現有程式碼連續
- E2 直接上 OR-Tools CP-SAT（Phase 3 原規劃）：全域最佳解、約束表達力強（連續性、跨店通勤），但複雜度高、不易解釋「為什麼這樣排」

### F. 同日跨店限制（PLAN 4.2 約束 5，從未實作）

- F1 同一天只能在一間門市上班（簡單、避免通勤問題）
- F2 允許同日跨店但班次間隔 ≥ N 小時
- F3 暫不限制（現況）

### G. 跨店範圍限制（哪些門市之間允許跨店）

並非組織內所有門市都允許互相支援，需要可設定的「跨店範圍」：

- G1 **跨店群組（cluster）**：每間門市屬於至多一個跨店群組（如 `Store.cross_group_id`），同群組內可互相跨店，未設群組 = 不參與跨店（只排自己店的員工）。簡單、可遞移、好理解
- G2 **成對白名單（pairwise）**：獨立表記錄「門市A ↔ 門市B 可跨店」，彈性最大（可表達 A-B、B-C 可跨但 A-C 不可），但設定 UI 與排班檢查都較複雜
- G3 **不限制**：組織內全部互通（原 A1 預設）

## ✅ 已決策（2026-06-11，2026-06-12 修訂）

**A1 + B2 + D（只重排 draft）+ E1 + F3（跨店間隔無限制）+ G1（跨店群組）**：

1. **A1 全組織聯合排班**：一次觸發排該組織當週所有門市；跨店互斥內建（同一員工同一時段只能在一店）
2. **B2 覆蓋率優先**：以全組織總缺口最小化為目標，偏好比重作 tie-breaker（比重高者優先入選；個人可能被派到比重較低但較缺人的店）
3. **D 只重排 draft**：published / archived 班表視為固定佔用不變動；`is_manual = true` 指派一律保留並佔用時段
4. **E1 greedy 擴展**：現有 day×hour 迴圈外移為組織層級，逐時段對「員工×缺人門市」配對分配；Phase 3 再換 CP-SAT
5. **F3 跨店間隔無限制**（2026-06-12 修訂，原 F2 ≥ 1 小時）：允許同日跨店，不同門市班次之間不設間隔要求（同店同時段互斥仍由跨店互斥保證）
6. **G1 跨店群組**（2026-06-12 新增）：只有同一跨店群組內的門市可以互相跨店排班。`Store.cross_group`（nullable 字串標籤，同標籤 = 同群組）；以員工的 `home_store_id`（所屬門市）為錨點——候選門市 = 角色群組涵蓋範圍 ∩（所屬門市 + 與所屬門市同群組的門市）；所屬門市未設群組 → 員工只會被排進所屬門市；員工未設所屬門市 → 沿用角色群組涵蓋範圍（浮動人力，不受群組限制）

## H. 每日排班上限（H1，2026-06-15 新增決策）

聯合排班後一名員工一天可能被多家門市瓜分時段，原本固定的 `DAILY_HOUR_MAX = 8` 寫死在 `scheduler.py`，無法依員工調整。

### ✅ 已決策（2026-06-15）

- **H1 per-employee 上限**：新增 `User.daily_hour_max`（nullable int，1–24）。`NULL` = 套用系統預設（`scheduler.DAILY_HOUR_MAX = 8`）；員工可在「偏好」分頁自行設定，覆蓋預設值
- 上限為**跨門市總和**：`run_greedy_org` 以 `daily_caps[uid] = users[uid].daily_hour_max or DAILY_HOUR_MAX` 計算同一天在所有門市的小時數總和，不分店分別計算（與跨店互斥邏輯一致，避免聯合排班把同一人塞滿多店）
- 員工自助設定，不需管理者權限（與門市偏好比重同一頁、同一存檔動作）
