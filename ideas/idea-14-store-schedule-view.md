## 核心問題：班表應有兩個入口

班表檢視應該要有兩個入口：

1. **排班人員入口（管理）** — 手動排班、發布班表、管理班表都在此
   - 只有具備對應權限者可進入
2. **員工入口（檢視）** — 只能看自己有權限看到的班表
   - 保留原本的圖表檢視
   - 新增類似日曆的週檢視模式

## 背景：現況其實是「一頁兩用、且完全沒分權」

現行 `/schedules` 一頁同時承擔了兩種角色，但**沒有任何權限把關**：

- 側邊欄「我的班表」（個人群組）與「班表管理」（排班管理群組）**指向同一個 `/schedules` 頁**。
- 該頁把「產生 / 發布 / 封存 / 跨日框選手動排班」直接顯示給**所有**到達者——一般員工點「我的班表」進來，也看得到（甚至能按）發布、產生等管理操作。
- 員工視角目前只有：員工 × 7×24 Grid + 覆蓋率熱力圖 + iCal 訂閱。沒有「只看自己」、沒有分色、沒有日曆週檢視。

所以本提案的骨幹是：**把這一頁拆成「管理入口（可編輯、全功能）」與「員工入口（唯讀、只看自己有權看的）」兩種體驗**，並補上權限門檻。

---

## A. 兩入口怎麼落地

- **A1（建議）同一路由、依權限切換體驗**：`/schedules` 內部依使用者權限決定渲染「管理版」（含產生/發布/封存/手動排班的全功能 Grid）或「員工版」（唯讀，預設只看自己）。側邊欄兩個連結仍可並存，但無管理權者即使從「班表管理」進來也只拿到唯讀版。維護一套頁面、最小新增面。
- **A2 兩個獨立路由**：管理入口維持 `/schedules`（受權限保護），員工入口另開 `/my-schedule`（唯讀、預設自己）。語意清楚、員工版可大幅簡化，但要維護兩頁與部分重複的 Grid。
- **A3 員工版併入既有頁面**：員工版作為 `/availability` 或個人頁的延伸分頁，不獨立。

> 待決策 A：拆成兩條路由（A2）還是同一頁依權限切換（A1）？

## B. 員工入口的檢視模式

PLAN §5.3.2 要的是「依門市分色的週表 + 日曆訂閱」。草稿要求「保留原圖表 + 新增日曆週檢視」。

- **B1（建議）兩種模式可切換**：
  - 「圖表」模式 = 保留現行員工 Grid / 熱力圖呈現
  - 「週曆」模式 = 列為門市、欄為週一～週日，格內顯示時段（如 `15:00–23:00`），**依 `Store.color` 分色**（對應 §5.3.2 表格）
- **B2 只做週曆**取代舊圖表（風險：失去既有熱力圖視角）。
- **B3 只保留現行圖表**，日曆延後。

> 待決策 B-i：員工版預設落在哪個模式（建議「週曆」較貼近員工心智）？
> 待決策 B-ii：分色依 `Store.color`；同一格跨多店時段如何呈現（並列色塊 / 分行）？

## C. 員工能看到的「範圍」

「只能檢視自己有權限能看到的班表」需要明確化：

- **C1（建議）預設只看自己**：員工版預設只顯示「我的班次」（跨門市彙整，沿用 §5.3.2／現有 iCal 同一份資料）。
- **C2 可看所屬門市的整張班表**：員工除了自己，還能看同店同事的班（唯讀），方便換班協調。是否開放、用哪個權限控制需定。
- **C3 兩者皆有**：預設只看自己，另以權限（如 `store.schedule.view`）解鎖「看整店」。

> 待決策 C：員工是否能看到同事/整店班表？若可，綁哪個權限？

## D. 管理入口的「跨身份組員工彙總」（§5.3.3 item 3 之一）

門市總覽 Grid 應「以門市為中心彙總跨身份組員工」。需確認現行 Grid 列出的員工範圍：

- **D1（建議）列出當週在該店有班 + 該店可排班範圍內的所有員工**，不分身份組（含跨店浮動員工、`cross_group` 支援者）。
- **D2 只列 home_store = 本店者**，另區塊標示「跨店支援」。
- **D3 維持現狀**（若現行已等同 D1，僅需確認）。

> 待決策 D-i：跨店支援／浮動員工要不要在 Grid 視覺區隔（不同底色／標籤）？

## E. 能力需求疊加熱力圖（§5.3.3 item 3 之二，僅管理入口）

資料已存在：`StoreSkillDemand.slots` 是 `boolean[7][24]`（某時段是否需要某技能），與人力需求 `DemandTemplate.slots`（`[7][24]` 人數）分屬兩表。

- **E1（建議）覆蓋率為底色 + 能力需求為角標**：沿用 `coverageColor` 當格子底色，缺技能時段加小標記（icon／邊框／角點），hover 顯示「缺哪個技能」。
- **E2 切換式雙圖層**：熱力圖分頁加 toggle，在「人力覆蓋率」與「能力需求覆蓋率」著色間切換。
- **E3 新增第三分頁**：「班表 / 覆蓋率 / 能力需求」。

> 待決策 E-i：能力需求「達成」如何判定？需員工技能資料——即該格排到的人是否有人具備所需技能標籤。要確認排班 Grid 端能否取得員工技能（員工管理 5.3.1 有「技能」tab，資料應已存在），及後端要不要新端點回傳「每格缺哪些技能」，或前端用既有資料現算。
> 待決策 E-ii：只在「有人力需求」的格子才判能力需求，還是獨立判定？

## F. 權限門檻

現行 `/schedules` 完全無門檻，需補上：

- 進入「管理版／可編輯」綁哪個權限？候選：`store.schedule.edit`、`org.schedule.arrange`、`org.schedule.view_all`（側邊欄 `showManager` 已用這組判斷顯示「班表管理」）。
- 缺管理權 → 落到員工唯讀版（隱藏產生/發布/封存/框選）。
- 員工唯讀版是否需要任何權限，或所有登入者皆可看自己（如同 `GET /users/me/payroll` 對所有人開放）。

> 待決策 F：唯讀／可編輯分界綁哪個權限；員工版是否完全免權限（看自己）。

## G. 門市負責人（`Store.manager_user_id`）連動

已有 `manager_user_id` 欄位但未連動任何權限/行為。

- **G1（建議）僅作預設選店 + 顯示**：負責人進入管理入口時預設選到自己負責的店；不額外給權限（權限仍走 F）。
- **G2 隱含檢視權**：身為某店 `manager_user_id` 即自動獲得該店 `store.schedule.view`（即使未指派身份組）。需改權限解析，影響面較大。
- **G3 不連動**：純 metadata，本次不處理。

> 待決策 G：負責人要不要「自動可看自己店班表」的隱含權限？
